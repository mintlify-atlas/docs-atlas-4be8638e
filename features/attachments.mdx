The Kuma attachments system provides permanent redirects for legacy file attachments that were historically uploaded to MDN's wiki, ensuring these URLs continue to work throughout the documentation.

## Overview

In the early days of MDN, users could upload files (images, fonts, PDFs, etc.) directly to wiki pages. These files were stored on disk and later migrated to S3 storage. The attachments system ensures these historical URLs remain functional.

<Info>
File uploads are no longer supported. The attachment system is read-only and serves only to redirect existing attachment URLs.
</Info>

## Architecture

The system uses JSON mapping files to redirect attachment URLs to their final S3 destinations without requiring a database.

### Redirect Mappings

Two JSON files provide attachment ID to S3 URL mappings:

1. **redirects.json** (kuma/attachments/redirects.json)
   - Maps Kuma attachment IDs to S3 paths
   - Used for standard file URLs like `/files/{id}/{filename}`

2. **mindtouch_redirects.json** (kuma/attachments/mindtouch_redirects.json)
   - Maps legacy MindTouch file IDs to S3 paths
   - Used for old MindTouch URLs like `/@api/deki/files/{id}/={filename}`

### How Mappings Were Created

The attachment mappings were generated by:

1. Scanning all mdn/content and mdn/translated-content repositories
2. Finding all attachment URLs referenced in documentation
3. Looking up each URL's final S3 destination
4. Creating a comprehensive mapping file

<Note>
Attachments not referenced in content or translated-content are not included in the mapping files.
Unmapped attachments return 404 errors.
</Note>

## URL Patterns

### Kuma File URLs

Format: `/files/{attachment_id}/{filename}`

Example:
```
/files/16014/LeagueMonoVariable.ttf
```

**Handler**: `raw_file()` view (kuma/attachments/views.py:52)

### MindTouch File URLs

Format: `/@api/deki/files/{file_id}/={filename}`

Example:
```
/@api/deki/files/5695/=alpha.png
```

**Handler**: `mindtouch_file_redirect()` view (kuma/attachments/views.py:66)

## Redirect Logic

The attachment views implement a two-tier redirect strategy based on the request hostname.

### View Implementation (kuma/attachments/views.py:52)

```python
def raw_file(request, attachment_id, filename):
    if attachment_id not in _redirects:
        return HttpResponseNotFound(f"{attachment_id} not a known redirect")
    
    if settings.DOMAIN in request.get_host():
        # Redirect to custom attachment URL first
        file_url = full_attachment_url(attachment_id, filename)
        return redirect(file_url, permanent=True)
    
    # Final redirect to S3
    return _redirect_final_path(_redirects[attachment_id])
```

### Redirect Flow

<Card title="Two-Stage Redirect" icon="arrows-turn-right">
  1. **Primary Domain**: Redirects to a canonical attachment URL on the main domain
  2. **CDN/Other Domains**: Redirects directly to the final S3 URL
</Card>

This approach:
- Provides consistent canonical URLs for SEO
- Allows CDN caching of redirects
- Enables future flexibility in attachment hosting

## S3 Integration

### Final URL Construction (kuma/attachments/views.py:78)

```python
def _redirect_final_path(final_path):
    redirect_url = "https://" if settings.ATTACHMENTS_AWS_S3_SECURE_URLS else "http://"
    redirect_url += settings.ATTACHMENTS_AWS_S3_CUSTOM_DOMAIN
    redirect_url += final_path
    
    response = redirect(redirect_url)
    
    # Set old Last-Modified header
    olden = datetime.datetime.now() - datetime.timedelta(days=365)
    response["Last-Modified"] = convert_to_http_date(olden)
    response["X-Frame-Options"] = f"ALLOW-FROM {settings.DOMAIN}"
    
    return response
```

### Configuration Settings

```python
# S3 bucket domain for attachments
ATTACHMENTS_AWS_S3_CUSTOM_DOMAIN = "media.prod.mdn.mozit.cloud"

# Use HTTPS for S3 URLs
ATTACHMENTS_AWS_S3_SECURE_URLS = True

# Main domain for canonical redirects
DOMAIN = "developer.mozilla.org"
```

## Response Headers

### Last-Modified Header

All attachment redirects include a `Last-Modified` header set to 365 days in the past:

```python
olden = datetime.datetime.now() - datetime.timedelta(days=365)
response["Last-Modified"] = convert_to_http_date(olden)
```

This signals to browsers and CDNs that these are stable, legacy resources suitable for aggressive caching.

### X-Frame-Options

Attachments can be embedded in iframes from the main MDN domain:

```python
response["X-Frame-Options"] = f"ALLOW-FROM {settings.DOMAIN}"
```

This allows documentation pages to embed attachment images and media while preventing third-party framing.

## Error Handling

### Unknown Attachment IDs

When an attachment ID is not found in the mapping files:

```python
if attachment_id not in _redirects:
    return HttpResponseNotFound(f"{attachment_id} not a known redirect")
```

This returns a 404 with a descriptive error message.

<Info>
Unmapped attachments are typically typos in URLs or files that were never referenced in published documentation.
</Info>

## Utility Functions

The attachments module includes helper functions for URL generation:

### full_attachment_url()

Generates the canonical attachment URL:

```python
def full_attachment_url(attachment_id, filename):
    return f"https://{settings.DOMAIN}/files/{attachment_id}/{filename}"
```

### full_mindtouch_attachment_url()

Generates the canonical MindTouch URL:

```python
def full_mindtouch_attachment_url(file_id, filename):
    return f"https://{settings.DOMAIN}/@api/deki/files/{file_id}/={filename}"
```

### convert_to_http_date()

Converts Python datetime to HTTP date format:

```python
def convert_to_http_date(dt):
    return dt.strftime("%a, %d %b %Y %H:%M:%S GMT")
```

## Historical Context

The evolution of MDN's attachment system:

1. **MindTouch Era**: Files stored in MindTouch CMS with URLs like `/@api/deki/files/`
2. **Kuma Era**: Files stored on disk with URLs like `/files/`
3. **S3 Migration**: All files moved to S3 for scalability and reliability
4. **Content Migration**: MDN moved from wiki to Git-based workflow (Yari)
5. **Redirect System**: JSON mapping created to maintain URL stability

<Card title="Design Decision" icon="lightbulb">
  The redirect mapping approach was chosen over database queries to:
  - Simplify deployment (no database required for attachments)
  - Improve performance (in-memory JSON lookups)
  - Enable easy updates (regenerate JSON files as needed)
</Card>

## Maintenance

### Updating Mappings

If new attachment references are discovered:

1. Look up the attachment's final S3 URL
2. Add the mapping to `redirects.json` or `mindtouch_redirects.json`
3. Deploy the updated JSON files
4. No code changes required

### Monitoring

404 errors for attachments indicate:
- Typos in documentation URLs
- References to non-existent files
- Attachments that need to be added to mappings
